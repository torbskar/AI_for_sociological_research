## Initial concept development

### Decisions

Paper type: methodological workflow paper, softly normative stance. Chosen over purely descriptive or strongly normative because the discipline needs practical guidance grounded in argument, not either a tutorial or an epistemological treatise.

Target journal: Methodological Innovations (Sage, open access). Selected from a shortlist on grounds of disciplinary fit and publication turnaround.

Core conceptual distinction: structured vs. unstructured AI use — not AI use vs. non-use — as the epistemically relevant axis for policy and methodological discussion.

File structure: adopted the folder layout specified in CLAUDE.md (logs/, literature/, draft/, supplementary/). Reflects existing practice, not a novel AI suggestion.

Logging convention: one daily log file and one daily author-input file. Author-input file records intellectual provenance of ideas; decision log records outcomes. Multiple work stages within a day are recorded as sections in the same file. Rationale: the paper's argument about transparency requires the paper's own production to be held to the same standard.

Workflow sequencing: literature search before drafting. When an outline and abstract were produced in an early exchange, the author redirected to search and file setup. This is a methodological commitment, not a procedural accident.

## Search and screening protocol

### Decisions

File structure: created from the spec in CLAUDE.md. The project directory was flat; all files moved to correct subdirectories and stub files created for all planned outputs.

Export format: CSV adopted over Scopus plaintext (.txt) for all future database exports. Rationale: structured columns allow systematic processing; output can be sorted in a spreadsheet; downstream import to reference manager and NotebookLM is cleaner.

Haiku for Phase 1 screening: adequate for classification against explicit criteria; economical for bulk records (~12–15K tokens per 300-record export); false negatives recoverable by spot-check; false positives caught at Phase 2.

Three-stage screening protocol: Haiku coarse screening → manual spot-check of excluded records → Sonnet relevance assessment. Sonnet targeted at A1, A2, and A3 only — the thin-coverage arguments where nuanced judgement is most needed and Haiku is least reliable. C4 and A5 adequately handled by Haiku. Spot-check covers ~10% of excluded records, focusing on those with generic exclusion reasons. Protocol documented in supplementary/boolean-search-guide.md.

Naming convention: all database export and screening output files prefixed with the search string letter from literature/boolean-searches.md (A, B, C for Block 1; D, E, F, G for Block 2). Pattern: [String]_[database]_export_[YYYY-MM-DD]_[uuid].csv, with _screened and _retained suffixes. Existing A_ outputs renamed to conform.

lit-screen skill: created and installed alongside other personal skills. Encodes the two-phase protocol, naming convention, inclusion/exclusion criteria, and argument codes. Enables short invocations ("screen the B export", "phase 2 on B"). Skill chosen over CLAUDE.md entry on grounds of modularity and reusability.

Logging format: one daily file per date with sections, replacing the former one-file-per-topic approach. CLAUDE.md updated accordingly.

String B exported: search string B (methodology focus) executed in Scopus; results exported as B_scopus_export_Apr 4-2026_[uuid].csv. Screening pending.
